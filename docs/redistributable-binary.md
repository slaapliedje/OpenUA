# Building a redistributable binary

A normal `make` build **embeds copyrighted FRUA bytes** and is therefore not
redistributable. `make release` (or `make NOEMBED=1`) builds a binary that
contains only the port's own code, so it *can* be shared — the user supplies
their own game data at runtime, exactly as they already must to play.

## What was embedded, and why

The Mac runtime initialises its A5 "world" (the global data + bss segment) from
a compressed `DATA` resource plus a `DREL` relocation table. The port replays
that at startup (`src/engine/data_pool_replay.c`). By default the build bakes
those tables into the binary as C arrays (`src/engine/data_pool.c`, generated by
`tools/dataemit.py` from `frua.rsc`) — **~31 KB of copyrighted FRUA
application bytes** (`g_a5_init_bytes` + 1026 relocations). That is the only
copyrighted payload the binary ever contained; everything else (the string
pool, art, wall tiles, module/design data) is already loaded from the user's
`frua.rsc` and design files at runtime, never compiled in.

## How `NOEMBED` removes it

`frua.rsc` — which the user supplies at runtime — *already contains* the
`DATA`, `ZERO`, and `DREL` resources the embedded tables were derived from. So
the binary doesn't need to carry them: it can rebuild the A5 world from
`frua.rsc` at launch.

- `make NOEMBED=1` emits a **stub** `data_pool.c` (zero entries — no copyrighted
  bytes).
- At startup, `data_pool_replay()` sees the stub and calls `dp_replay_runtime()`
  instead: it `GetResource`s `DATA`/`ZERO`/`DREL` from `frua.rsc`, expands the
  compressed image with the `ZERO` run-length table, and applies the `DREL`
  relocations — the identical decode `tools/datapool.py` does at build time,
  ported to portable C in `src/engine/data_pool_decode.h`.

The C decode is verified **bit-for-bit** against the Python reference by
`tests/test_data_pool_decode.py` (a synthetic case that always runs, plus the
real `DATA`/`ZERO`/`DREL` from `frua.rsc` when present — the latter reproduces
the 31336-byte image exactly). The stub build boots to the menu with the real
title/version strings sourced from the runtime-reconstructed A5 world, identical
to the embedded build.

## `make release`

`make release` does all of it: `NOEMBED=1`, a **stripped** link (drops the
~3 MB symbol table — a `.prg` goes from ~4.3 MB to ~1 MB on disk), and it
**does not bundle `frua.rsc`** (that is the user's copyrighted data). The
`dist/` payload is the port binary, README, and release notes only.

```sh
make release VERSION=0.2.0-beta      # -> dist/openua-falcon-<ver>.zip
```

The one FRUA-identifying literal still in the binary is the string
`"Unlimited Adventures"` — a **port-authored fallback** used only if the
runtime replay can't supply the real title (a trademark/product-name reference,
not copyrighted game content). With data present, the real title/version load
from `frua.rsc`.

## Caveats

- The binary still needs the user's `frua.rsc` (and design/data files) at
  runtime — it plays nothing on its own. This changes *redistributability of the
  binary*, not the requirement that each player own FRUA.
- The Amiga build has the same knob (`make MACHINE=amiga NOEMBED=1`); the
  runtime replay path is shared engine code. A dedicated Amiga release target is
  future work.
- Emulator-validated only — see the top-level README's hardware warning.
